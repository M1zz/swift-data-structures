Array
Q01

Swift Array의 Copy-on-Write(CoW) 메커니즘을 설명해 주세요.

모범 답안

Swift의 Array는 값 타입(struct)이지만 내부적으로 힙에 할당된 버퍼를 공유합니다. 변수를 복사할 때 즉시 메모리를 복제하지 않고, 참조 카운트만 증가시킵니다. 실제로 한쪽에서 mutating 연산(append, 요소 변경 등)이 발생하면 그때 비로소 버퍼를 복사합니다. 이를 통해 불필요한 복사를 줄이면서도 값 의미론을 유지합니다. isKnownUniquelyReferenced 함수로 참조가 유일한지 확인하여 CoW 트리거 여부를 결정합니다.

꼬리 질문: CoW 때문에 예상치 못하게 성능이 저하되는 시나리오는 어떤 것이 있나요?

Q02

Array와 ContiguousArray의 차이는 무엇인가요?

모범 답안

Array는 Element가 클래스 타입이거나 @objc 프로토콜일 때 내부적으로 NSArray와 브리징될 수 있어, 요소를 간접 참조(boxing)할 가능성이 있습니다. 반면 ContiguousArray는 항상 연속된 메모리 블록에 요소를 저장하므로 Objective-C 브리징이 필요 없는 순수 Swift 코드에서 클래스 타입 배열의 성능이 더 좋습니다. 값 타입 Element에서는 둘의 성능 차이가 거의 없습니다. 일반적으로 Objective-C 상호운용이 필요 없다면 클래스 배열에는 ContiguousArray를 권장합니다.

꼬리 질문: ArraySlice와 ContiguousArray를 함께 사용할 때 주의할 점은?

Q03

Array의 append가 amortized O(1)인 이유를 설명하세요.

모범 답안

Array는 내부 버퍼 용량이 부족하면 기존 용량의 2배(기하급수적 성장)로 새 버퍼를 할당하고 기존 요소를 복사합니다. 개별 append는 버퍼 재할당 시 O(n)이 걸리지만, n번의 append에 걸쳐 재할당 횟수가 log(n)번뿐이므로 총 비용을 n으로 나누면 평균 O(1)이 됩니다. 이것이 상각(amortized) 분석입니다. reserveCapacity(_:)를 사용하면 미리 충분한 버퍼를 확보하여 재할당을 아예 방지할 수 있습니다.

꼬리 질문: reserveCapacity를 호출하면 항상 성능이 좋아지나요? 오히려 해가 되는 경우는?

Q04

ArraySlice의 메모리 특성과 주의할 점을 설명하세요.

모범 답안

ArraySlice는 원본 Array의 버퍼를 공유하며 시작/끝 인덱스만 다르게 가리킵니다. 따라서 슬라이스를 생성해도 복사가 일어나지 않아 O(1)입니다. 하지만 원본 Array가 해제되어도 슬라이스가 살아 있으면 전체 버퍼가 메모리에 유지됩니다. 큰 배열에서 작은 슬라이스만 오래 보관하면 메모리 누수와 유사한 상황이 발생합니다. 이를 방지하려면 Array(slice)로 독립 복사본을 만들어야 합니다. 또한 슬라이스의 startIndex가 0이 아닐 수 있어 인덱스 사용에 주의해야 합니다.

꼬리 질문: Substring과 ArraySlice의 메모리 공유 방식이 유사한데, 이러한 패턴의 장단점은?

LinkedList
Q05

LinkedList의 Array 대비 장단점을 설명하세요.

모범 답안

LinkedList는 중간 삽입/삭제가 O(1)이고(노드 참조가 있을 때), 크기가 동적이라 재할당이 없습니다. 반면 Array는 연속 메모리 덕분에 캐시 지역성이 우수하고 인덱스 접근이 O(1)입니다. LinkedList는 노드마다 포인터 오버헤드가 있고, 랜덤 접근이 O(n)이며, 캐시 미스가 빈번합니다. 실무에서는 대부분 Array가 유리하며, 빈번한 head/tail 삽입삭제가 핵심인 경우(예: LRU 캐시)에만 LinkedList를 고려합니다.

꼬리 질문: Swift 표준 라이브러리에 LinkedList가 없는 이유는 무엇일까요?

Q06

LinkedList의 노드를 class로 구현하는 이유는 무엇인가요?

모범 답안

LinkedList 노드는 다른 노드를 참조(pointing)해야 하므로 참조 타입인 class가 적합합니다. struct로 구현하면 값 타입이라 노드를 변수에 할당할 때마다 복사가 발생하고, 재귀적 구조(자기 자신을 프로퍼티로 갖는 것)가 불가능합니다. class는 힙에 할당되어 여러 곳에서 동일 노드를 참조할 수 있고, next/prev 포인터로 노드 간 연결을 자연스럽게 표현합니다. 다만 ARC 오버헤드와 순환 참조에 주의해야 합니다.

꼬리 질문: enum의 indirect case로 LinkedList를 구현하면 어떤 차이가 있나요?

Q07

Singly LinkedList와 Doubly LinkedList의 차이와 각각의 사용처를 말해주세요.

모범 답안

Singly LinkedList는 각 노드가 next 포인터만 갖고, Doubly LinkedList는 next와 prev를 모두 갖습니다. Singly는 메모리가 적게 들지만 역방향 순회나 이전 노드 삭제가 O(n)입니다. Doubly는 양방향 순회와 임의 노드 삭제가 O(1)이지만 포인터 하나가 추가됩니다. Singly는 스택, 해시 체이닝 등에, Doubly는 LRU 캐시, 브라우저 히스토리, NSCache 내부 등에 쓰입니다.

꼬리 질문: Doubly LinkedList에서 순환 참조를 방지하려면 어떻게 해야 하나요?

Q08

LinkedList에서 메모리 누수가 발생하는 원인과 해결법을 설명하세요.

모범 답안

Doubly LinkedList에서 prev와 next가 모두 strong 참조이면, 두 노드가 서로를 가리켜 retain cycle이 생깁니다. ARC는 순환 참조를 자동 해제하지 못하므로 메모리 누수가 발생합니다. 해결법으로는 prev를 weak로 선언하거나, 리스트 해제 시 모든 노드의 연결을 명시적으로 끊는 removeAll()을 구현하는 방법이 있습니다. 또한 리스트 자체를 class로 감싸고 deinit에서 순회하며 연결을 해제하는 패턴도 자주 사용됩니다.

꼬리 질문: Instruments의 Leaks 도구로 이러한 순환 참조를 어떻게 탐지하나요?

Stack / Queue
Q09

Stack과 Queue의 차이를 설명하고, 각각의 실무 활용 예시를 들어주세요.

모범 답안

Stack은 LIFO(Last In, First Out), Queue는 FIFO(First In, First Out) 구조입니다. Stack은 Navigation 뒤로가기(UINavigationController의 viewControllers), Undo/Redo, 재귀 호출 스택, 괄호 매칭 등에 사용됩니다. Queue는 네트워크 요청 대기열, GCD의 DispatchQueue, 이벤트 처리 순서 보장, BFS 탐색 등에 사용됩니다. Swift에서 Stack은 Array의 append/popLast로 간단히 구현하고, Queue는 Array의 removeFirst가 O(n)이므로 별도 최적화가 필요합니다.

꼬리 질문: Queue를 Array로 구현할 때 removeFirst()의 O(n) 문제를 어떻게 해결하나요?

Q10

Stack 2개로 Queue를 구현하는 방법을 설명하세요.

모범 답안

inStack과 outStack 두 개를 사용합니다. enqueue 시 inStack에 push하고, dequeue 시 outStack이 비어있으면 inStack의 모든 요소를 pop하여 outStack에 push(순서 뒤집기)한 뒤 outStack에서 pop합니다. outStack에 요소가 있으면 바로 pop합니다. 이렇게 하면 각 요소가 최대 2번 이동하므로 amortized O(1)의 enqueue/dequeue가 보장됩니다. 이 기법은 함수형 프로그래밍에서도 자주 등장하는 패턴입니다.

꼬리 질문: 이 구현에서 peek 연산의 시간 복잡도는 어떻게 되나요?

Q11

OperationQueue와 DispatchQueue의 차이를 설명하세요.

모범 답안

DispatchQueue는 GCD 기반으로 클로저 단위의 경량 작업을 처리하며, serial/concurrent 큐를 지원합니다. OperationQueue는 Operation 객체를 사용하여 작업 간 의존성(dependency) 설정, 취소(cancel), KVO 기반 상태 관찰, 최대 동시 실행 수(maxConcurrentOperationCount) 제어가 가능합니다. 단순 비동기 작업은 DispatchQueue, 복잡한 작업 흐름 관리가 필요하면 OperationQueue를 선택합니다. OperationQueue도 내부적으로 GCD를 사용합니다.

꼬리 질문: Swift Concurrency(async/await)가 등장한 후에도 OperationQueue가 필요한 경우는?

Q12

Stack Overflow가 발생하는 원인과 iOS에서의 대처법을 설명하세요.

모범 답안

Stack Overflow는 재귀 호출이 너무 깊어 호출 스택 메모리(iOS 기본 약 1MB)를 초과할 때 발생합니다. 주요 원인은 탈출 조건 없는 재귀, 매우 깊은 트리 순회, 상호 재귀 등입니다. 대처법으로는 재귀를 반복문으로 변환(명시적 스택 사용), 꼬리 재귀 최적화(Swift 컴파일러가 보장하지는 않음), 재귀 깊이 제한 설정 등이 있습니다. Thread의 stackSize를 늘리는 것은 근본 해결이 아니며, 알고리즘 자체를 반복적으로 재설계하는 것이 바람직합니다.

꼬리 질문: Swift 컴파일러가 꼬리 재귀 최적화를 수행하는 조건은 무엇인가요?

Deque
Q13

Deque를 Ring Buffer로 구현하는 이유를 설명하세요.

모범 답안

Ring Buffer(원형 버퍼)는 고정 크기 배열에서 head와 tail 인덱스를 모듈로 연산으로 순환시켜, 앞뒤 삽입/삭제 모두 O(1)을 달성합니다. 일반 배열은 앞쪽 삽입/삭제 시 요소를 밀어야 하므로 O(n)이지만, Ring Buffer는 인덱스만 이동하면 됩니다. 연속된 메모리를 사용하므로 캐시 지역성도 좋습니다. Swift의 swift-collections 패키지의 Deque도 이 방식을 사용합니다.

꼬리 질문: Ring Buffer가 가득 찼을 때 어떻게 확장하나요?

Q14

Array 기반 Deque에서 발생할 수 있는 성능 문제를 설명하세요.

모범 답안

일반 Array로 Deque를 구현하면 removeFirst()와 insert(at: 0)가 모든 요소를 한 칸씩 이동시켜 O(n)입니다. n개 요소를 순차적으로 dequeue하면 O(n^2)가 됩니다. 또한 앞쪽에서 반복 삭제하면 버퍼 앞쪽에 빈 공간이 낭비됩니다. 이를 해결하려면 Ring Buffer를 사용하거나, head 오프셋을 두어 논리적으로만 제거하고 주기적으로 compaction하는 방법을 쓸 수 있습니다.

꼬리 질문: Swift Array의 removeFirst()가 내부적으로 최적화되어 있는 경우가 있나요?

Q15

Deque의 실무 활용 사례를 말해주세요.

모범 답안

Deque는 양쪽에서 삽입/삭제가 필요한 곳에 활용됩니다. 슬라이딩 윈도우 알고리즘에서 최대/최소값을 효율적으로 추적할 때, 작업 스케줄러에서 우선순위 높은 작업을 앞에 삽입(work stealing), BFS의 변형인 0-1 BFS, 브라우저의 앞으로/뒤로 히스토리 관리, 최근 항목 N개 유지(채팅 메시지 버퍼) 등에 사용됩니다. Apple의 swift-collections 패키지에서 Deque를 공식 제공하고 있습니다.

꼬리 질문: 슬라이딩 윈도우 최댓값 문제에서 Deque로 O(n) 해법을 구현하는 방법은?

Q16

removeFirst()를 O(1)로 만드는 전략을 설명하세요.

모범 답안

첫 번째 전략은 Ring Buffer로, head 인덱스를 1 증가시키기만 하면 됩니다. 두 번째는 head 오프셋 방식으로, 실제 삭제 대신 offset 변수를 두고 논리적 시작점을 이동시킵니다. offset이 배열 크기의 절반을 넘으면 앞부분을 잘라내고 offset을 리셋합니다. 세 번째는 앞서 설명한 Two-Stack 방식으로, reversed stack에서 pop하면 O(1)입니다. swift-collections의 Deque는 Ring Buffer 기반이며, 세 가지 모두 amortized O(1) 또는 worst-case O(1)을 달성합니다.

꼬리 질문: worst-case O(1)과 amortized O(1)의 차이가 중요한 실무 상황은?

Binary Tree
Q17

이진 트리의 4가지 순회 방법과 각각의 사용처를 설명하세요.

모범 답안

전위(Pre-order): 루트→좌→우, 트리 복사/직렬화에 사용. 중위(In-order): 좌→루트→우, BST에서 정렬된 순서로 출력. 후위(Post-order): 좌→우→루트, 트리 삭제(자식 먼저 해제), 수식 트리 계산. 레벨(Level-order): BFS로 층별 순회, 트리 높이 계산, 최단 경로 탐색에 사용됩니다. 전위/중위/후위는 DFS 기반이고 재귀 또는 명시적 스택으로, 레벨 순회는 큐로 구현합니다.

꼬리 질문: 전위 순회와 중위 순회 결과만으로 원본 트리를 복원할 수 있는 이유는?

Q18

완전 이진 트리, 포화 이진 트리, 균형 이진 트리의 차이를 설명하세요.

모범 답안

완전(Complete) 이진 트리는 마지막 레벨을 제외하고 모든 레벨이 꽉 차 있고, 마지막 레벨은 왼쪽부터 채워집니다. Heap이 대표적입니다. 포화(Full/Perfect) 이진 트리는 모든 레벨이 완전히 채워진 트리로, 노드 수가 2^h - 1입니다. 균형(Balanced) 이진 트리는 좌우 서브트리 높이 차이가 1 이하인 트리로, AVL 트리가 대표적입니다. 균형 트리는 검색/삽입/삭제가 O(log n)을 보장합니다.

꼬리 질문: Red-Black 트리와 AVL 트리의 균형 유지 전략 차이는?

Q19

재귀 없이 이진 트리를 순회하는 방법을 설명하세요.

모범 답안

명시적 스택을 사용하여 반복문으로 순회할 수 있습니다. 중위 순회의 경우: 현재 노드를 왼쪽 끝까지 따라가며 스택에 push하고, 더 이상 왼쪽이 없으면 pop하여 방문 후 오른쪽으로 이동합니다. 전위 순회는 스택에서 pop한 노드를 먼저 방문하고 오른쪽, 왼쪽 순으로 push합니다. Morris Traversal은 스택 없이 O(1) 공간으로 순회하는 기법으로, 쓰레드(thread)를 이용해 임시로 트리 구조를 변경합니다.

꼬리 질문: Morris Traversal이 트리를 임시 변경하는데, 멀티스레드 환경에서 안전한가요?

Q20

모범 답안

Q25

Heap을 Array로 구현하는 이유와 인덱스 관계를 설명하세요.

모범 답안

Q26

heapify의 시간 복잡도가 O(n)인 이유를 설명하세요.

모범 답안

Bottom-up heapify는 배열의 뒤쪽(리프)부터 앞쪽(루트)으로 sift-down을 수행합니다. 직관적으로 n개 노드에 O(log n) sift-down이면 O(n log n) 같지만, 실제로는 대부분의 노드가 트리 하단에 있어 이동 거리가 짧습니다. 높이 h에서의 노드 수는 n/2^(h+1)이고 각각 h번 이동하므로, 총 비용은 sum(h * n/2^(h+1)) = O(n)입니다. 이는 top-down 방식(각 요소를 삽입하며 sift-up)의 O(n log n)보다 효율적입니다.

Top-K 문제에서 Min Heap을 사용하는 이유를 설명하세요.

모범 답안

n개 중 가장 큰 K개를 찾을 때, 크기 K의 Min Heap을 유지합니다. 새 요소가 Heap의 최솟값(루트)보다 크면 루트를 교체하고 sift-down합니다. 이렇게 하면 Heap에는 항상 지금까지 본 요소 중 가장 큰 K개가 남습니다. 시간 복잡도는 O(n log K)이고 공간은 O(K)입니다. Max Heap으로 전체를 넣으면 O(n log n) 공간 O(n)이 필요합니다. K가 n보다 훨씬 작을 때 Min Heap 방식이 메모리와 시간 모두 효율적입니다.

Swift Dictionary의 내부 구현을 설명하세요.

Q35

Load Factor와 Rehashing의 관계를 설명하세요.

모범 답안

Load Factor = 저장된 요소 수 / 버킷 수입니다. Load Factor가 높아지면 충돌이 빈번해져 성능이 O(1)에서 O(n)으로 저하됩니다. 이를 방지하기 위해 특정 임계값(Swift Dictionary는 약 75%)을 넘으면 버킷 수를 2배로 늘리고 모든 요소를 새 해시값으로 재배치(rehashing)합니다. Rehashing은 O(n) 연산이지만 빈도가 낮아 amortized O(1)을 유지합니다. Dictionary(minimumCapacity:)로 미리 크기를 지정하면 rehashing을 줄일 수 있습니다.

Q37

Trie의 검색 시간 복잡도가 O(m)인 이유를 설명하세요.

모범 답안

Trie와 Hash Table의 장단점을 비교하세요.

모범 답안
