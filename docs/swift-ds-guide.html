Chapter 01
Array vs ContiguousArray
Swift의 두 가지 배열 타입, 그 차이를 메모리 수준에서 이해한다
개념

Array는 Swift 표준 배열입니다. 내부적으로 Copy-on-Write(CoW) 를 사용하여 불필요한 복사를 줄입니다. 요소가 클래스 타입이면 NSArray 브리징이 가능하지만 그만큼 오버헤드가 있습니다.

ContiguousArray는 항상 메모리에 연속 배치됩니다. 클래스/프로토콜 타입 요소를 다룰 때 Array보다 빠르며, 브리징 오버헤드가 없습니다. 구조체(Int, Double 등)에서는 Array와 성능이 동일합니다.

핵심 포인트
  1. 구조체 원소라면 Array와 ContiguousArray 성능 동일
  2. 클래스 원소 + 성능 민감 → ContiguousArray 선택
  3. CoW 동작: 복사 후 쓰기 시점에 실제 메모리 복사 발생
  4. reserveCapacity(_:)로 재할당 횟수 최소화 가능
시간 복잡도
연산Array설명
인덱스 접근O(1)연속 메모리
appendO(1) amortizedcapacity 2배 확장
insert(at:)O(n)요소 이동 필요
remove(at:)O(n)요소 이동 필요
containsO(n)선형 탐색
생각해볼 것
함수에 배열을 인수로 전달하면 CoW는 언제 발동될까요? 단순 읽기와 쓰기 시점에 각각 무슨 일이 일어나는지 추적해보세요.
체크리스트
CoW 동작 원리를 직접 코드로 확인했다
Array와 ContiguousArray 각각으로 같은 코드를 작성해봤다
insert / remove의 O(n) 이유를 그림으로 설명할 수 있다
reserveCapacity가 성능에 미치는 영향을 이해했다
Chapter 02
Linked List
노드를 포인터로 연결 — Array로 할 수 없는 것을 가능하게
개념

연결 리스트는 Node 객체들이 next 포인터로 연결된 선형 자료구조입니다. Array와 달리 메모리가 연속하지 않아도 되므로, 중간 삽입/삭제가 O(1)(노드 위치를 알고 있을 때)입니다.

단방향(Singly)과 양방향(Doubly) 두 종류가 있으며, Swift에서는 class Node를 사용해 참조 의미론으로 구현합니다.

핵심 포인트
  1. head/tail 포인터 관리가 핵심 — nilチェック 주의
  2. 삽입/삭제는 포인터 2~3개만 바꾸면 됨
  3. 인덱스 접근은 O(n) — 중간 탐색은 느림
  4. Swift에서 weak 참조로 retain cycle 방지 (양방향)
시간 복잡도
연산단방향양방향
head 삽입/삭제O(1)O(1)
tail 삽입O(1)*O(1)
tail 삭제O(n)O(1)
중간 삽입/삭제O(1)**O(1)**
인덱스 접근O(n)O(n)

* tail 포인터 보유 시 / ** 해당 노드 참조를 이미 알고 있을 때

생각해볼 것
왜 Swift의 표준 라이브러리는 LinkedList를 제공하지 않을까요? Array의 CoW와 비교했을 때 실제 앱에서 LinkedList가 유리한 케이스는 무엇일까요?
체크리스트
head/tail을 직접 관리하는 LinkedList를 구현했다
삽입과 삭제 시 포인터 변경 순서를 이해했다
단방향 vs 양방향 트레이드오프를 설명할 수 있다
역순 출력, 중간값 찾기 등 기본 문제를 풀어봤다
Chapter 03
Stack / Queue
LIFO와 FIFO — 알고리즘의 기본 빌딩 블록
개념

Stack(스택)은 후입선출(LIFO) 구조입니다. push로 쌓고 pop으로 꺼냅니다. 함수 호출 스택, 되돌리기(Undo), 괄호 검증에 사용됩니다.

Queue(큐)는 선입선출(FIFO) 구조입니다. enqueue로 넣고 dequeue로 꺼냅니다. BFS, 작업 스케줄링, iOS 메시지 처리에 사용됩니다.

Swift에서는 protocol로 추상화하면 Array/LinkedList 등 다양한 내부 구현을 교체할 수 있습니다.

시간 복잡도
연산StackQueue (Array)Queue (LinkedList)
push / enqueueO(1)O(1) amortizedO(1)
pop / dequeueO(1)O(n)O(1)
peekO(1)O(1)O(1)

체크리스트
Stack으로 괄호 유효성 검사를 구현했다
두 Stack으로 Queue를 구현해봤다
BFS에서 Queue가 왜 필요한지 설명할 수 있다
Chapter 04
Deque
양쪽 모두 O(1) — 링 버퍼로 구현하는 Double-Ended Queue
개념

Deque(덱)은 앞(front)과 뒤(back) 양쪽에서 O(1)으로 삽입/삭제가 가능한 자료구조입니다. Stack과 Queue를 모두 대체할 수 있습니다.

링 버퍼(Circular Buffer)로 구현하면 고정 크기 배열에서 head/tail 인덱스만 이동하여 메모리 재할당 없이 O(1) 연산을 달성합니다. Swift의 Collections 패키지 Deque가 이 방식입니다.

시간 복잡도
연산링 버퍼 Deque
prepend / appendO(1) amortized
removeFirst / removeLastO(1)
인덱스 접근O(1)

링 버퍼가 꽉 찼을 때 어떻게 동작해야 할까요? 고정 크기(오래된 것 버리기) vs 동적 확장 중 어떤 전략이 언제 적합한가요?
체크리스트
링 버퍼 구조의 head/tail 인덱스 계산을 이해했다
Deque로 슬라이딩 윈도우 최대값 문제를 풀었다
Swift Collections 패키지의 Deque를 실제로 써봤다
Chapter 05
Binary Tree
전위 · 중위 · 후위 · 레벨 순회 — 트리의 모든 것
개념

이진 트리는 각 노드가 최대 2개의 자식(left, right)을 가지는 계층적 자료구조입니다. 재귀적 구조이기 때문에 재귀 알고리즘과 찰떡궁합입니다.

순회(Traversal) 4가지: 전위(Pre-order) 루트→좌→우, 중위(In-order) 좌→루트→우, 후위(Post-order) 좌→우→루트, 레벨 순서(BFS) 층별 탐색.

순회별 활용
  1. 전위 — 트리 복사, 직렬화
  2. 중위 — BST에서 정렬된 순서 출력
  3. 후위 — 하위 폴더 크기 계산, 메모리 해제
  4. 레벨 BFS — 최단 경로, 층별 출력
생각해볼 것
재귀 없이 스택을 사용하여 전위/중위 순회를 구현할 수 있나요? 두 가지 방법의 공간 복잡도를 비교해보세요.
체크리스트
4가지 순회를 재귀로 구현했다
중위 순회를 Stack 기반 반복문으로 구현했다
트리의 높이와 노드 수를 재귀로 구했다
Chapter 06
Binary Search Tree
삽입 · 탐색 · 삭제(3케이스) — 정렬된 트리의 힘
개념

BST는 모든 노드가 왼쪽 자식 < 현재 노드 < 오른쪽 자식 규칙을 만족하는 이진 트리입니다. 이 성질 덕분에 탐색이 절반씩 줄어들어 평균 O(log n)입니다.

삭제 3케이스: ① 자식 없음 — 그냥 삭제, ② 자식 1개 — 자식으로 대체, ③ 자식 2개 — 오른쪽 서브트리의 최솟값(Successor)으로 대체.

시간 복잡도
연산평균 (균형)최악 (편향)
탐색O(log n)O(n)
삽입O(log n)O(n)
삭제O(log n)O(n)
생각해볼 것
정렬된 배열을 순서대로 삽입하면 BST는 어떤 모양이 될까요? 이를 방지하기 위해 AVL Tree와 Red-Black Tree는 어떤 전략을 쓰나요?
체크리스트
삭제 3케이스를 구현하고 각각 테스트했다
중위 순회로 정렬된 배열이 나오는 것을 확인했다
편향 트리가 되는 케이스를 실험해봤다
Chapter 07
Heap / Priority Queue
siftUp · siftDown — 가장 크거나 작은 것을 항상 O(1)에
개념

Heap은 완전 이진 트리를 배열에 저장하는 자료구조입니다. Max Heap은 부모가 자식보다 항상 크거나 같고, Min Heap은 그 반대입니다.

인덱스 i의 부모: (i-1)/2, 왼쪽 자식: 2*i+1, 오른쪽 자식: 2*i+2.

Priority Queue는 Heap으로 구현됩니다. iOS의 OperationQueue.qualityOfService, Dijkstra 알고리즘에서 핵심 역할.

시간 복잡도
연산복잡도
최대/최소값 조회 (peek)O(1)
삽입 (siftUp)O(log n)
삭제 (siftDown)O(log n)
Heapify (배열→힙)O(n)

Heap Sort는 어떻게 O(n log n)을 달성하면서도 추가 메모리를 O(1)만 사용하나요? QuickSort와 비교했을 때 언제 Heap Sort가 더 나을까요?
체크리스트
siftUp과 siftDown을 직접 구현했다
Priority Queue로 K번째 최대값 문제를 풀었다

Chapter 08
Graph
인접행렬 vs 인접리스트 · BFS · DFS — 관계를 모델링하다
개념

그래프는 정점(Vertex)과 간선(Edge)으로 구성됩니다. 방향/무방향, 가중치 유무로 다양한 종류가 있습니다.

인접행렬: O(1) 간선 확인, O(V²) 공간 — 밀집 그래프에 유리. 인접리스트: O(degree) 간선 확인, O(V+E) 공간 — 희소 그래프에 유리.

BFS는 Queue 사용, 최단 경로(가중치 없는). DFS는 Stack/재귀 사용, 사이클 감지·위상정렬.

시간 복잡도
표현 방식공간간선 확인인접 목록
인접행렬O(V²)O(1)O(V)
인접리스트O(V+E)O(degree)O(degree)
체크리스트
BFS로 최단 경로(홉 수)를 구했다
DFS로 사이클 감지를 구현했다
두 표현 방식의 트레이드오프를 설명할 수 있다
Chapter 09
Hash Table
충돌 해결 · Swift Dictionary 내부 — O(1) 탐색의 비밀
개념

Hash Table은 키를 해시 함수로 인덱스로 변환하여 O(1) 삽입/탐색/삭제를 달성합니다.

충돌 해결: ① Chaining — 같은 버킷에 LinkedList 연결 (최악 O(n)). ② Open Addressing — 빈 슬롯을 찾아 이동 (캐시 친화적). Swift Dictionary는 Open Addressing + Robin Hood Hashing을 사용합니다.

Load Factor가 임계값(보통 0.75)을 넘으면 버킷을 2배로 늘리고 Rehashing합니다.

시간 복잡도
연산평균최악
탐색 / 삽입 / 삭제O(1)O(n)

Chaining 방식의 Hash Table을 직접 구현했다

Chapter 10
Trie
자동완성 DFS · O(m) 탐색 — 문자열 특화 트리
개념

Trie(트라이)는 문자열을 글자 단위로 트리에 저장하는 자료구조입니다. 검색 길이가 m이면 O(m)으로 탐색 가능 — 해시 충돌 없이 정확합니다.

각 노드는 자식 딕셔너리([Character: TrieNode])와 isEndOfWord 플래그를 가집니다. 자동완성은 prefix까지 내려간 뒤 DFS로 모든 단어를 수집합니다.

실제 사용처: iOS 키보드 자동완성, 검색 엔진 prefix 인덱싱, IP 라우팅 테이블.

시간 복잡도
연산복잡도비고
삽입O(m)m = 문자열 길이
탐색O(m)정확히 m 노드만 방문
자동완성O(m + k)k = 결과 수
공간O(알파벳 × 노드 수)영문 최대 26배
