Chapter 01
Array vs ContiguousArray
Swift의 두 가지 배열 타입, 그 차이를 메모리 수준에서 이해한다
개념

Array는 Swift 표준 배열입니다. 내부적으로 Copy-on-Write(CoW) 를 사용하여 불필요한 복사를 줄입니다. 요소가 클래스 타입이면 NSArray 브리징이 가능하지만 그만큼 오버헤드가 있습니다.

ContiguousArray는 항상 메모리에 연속 배치됩니다. 클래스/프로토콜 타입 요소를 다룰 때 Array보다 빠르며, 브리징 오버헤드가 없습니다. 구조체(Int, Double 등)에서는 Array와 성능이 동일합니다.

핵심 포인트
  1. 구조체 원소라면 Array와 ContiguousArray 성능 동일
  2. 클래스 원소 + 성능 민감 → ContiguousArray 선택
  3. CoW 동작: 복사 후 쓰기 시점에 실제 메모리 복사 발생
  4. reserveCapacity(_:)로 재할당 횟수 최소화 가능
시간 복잡도
연산Array설명
인덱스 접근O(1)연속 메모리
appendO(1) amortizedcapacity 2배 확장
insert(at:)O(n)요소 이동 필요
remove(at:)O(n)요소 이동 필요
containsO(n)선형 탐색

함수에 배열을 인수로 전달하면 CoW는 언제 발동될까요? 단순 읽기와 쓰기 시점에 각각 무슨 일이 일어나는지 추적해보세요.
체크리스트
CoW 동작 원리를 직접 코드로 확인했다
Array와 ContiguousArray 각각으로 같은 코드를 작성해봤다
insert / remove의 O(n) 이유를 그림으로 설명할 수 있다
reserveCapacity가 성능에 미치는 영향을 이해했다
Chapter 02
Linked List
노드를 포인터로 연결 — Array로 할 수 없는 것을 가능하게
개념

연결 리스트는 Node 객체들이 next 포인터로 연결된 선형 자료구조입니다. Array와 달리 메모리가 연속하지 않아도 되므로, 중간 삽입/삭제가 O(1)(노드 위치를 알고 있을 때)입니다.

단방향(Singly)과 양방향(Doubly) 두 종류가 있으며, Swift에서는 class Node를 사용해 참조 의미론으로 구현합니다.

핵심 포인트
  1. head/tail 포인터 관리가 핵심 — nilチェック 주의
  2. 삽입/삭제는 포인터 2~3개만 바꾸면 됨
  3. 인덱스 접근은 O(n) — 중간 탐색은 느림
  4. Swift에서 weak 참조로 retain cycle 방지 (양방향)
시간 복잡도
연산단방향양방향
head 삽입/삭제O(1)O(1)
tail 삽입O(1)*O(1)
tail 삭제O(n)O(1)
중간 삽입/삭제O(1)**O(1)**
인덱스 접근O(n)O(n)

* tail 포인터 보유 시 / ** 해당 노드 참조를 이미 알고 있을 때

체크리스트
head/tail을 직접 관리하는 LinkedList를 구현했다
삽입과 삭제 시 포인터 변경 순서를 이해했다
단방향 vs 양방향 트레이드오프를 설명할 수 있다
역순 출력, 중간값 찾기 등 기본 문제를 풀어봤다
Chapter 03
Stack / Queue
LIFO와 FIFO — 알고리즘의 기본 빌딩 블록
개념

Stack(스택)은 후입선출(LIFO) 구조입니다. push로 쌓고 pop으로 꺼냅니다. 함수 호출 스택, 되돌리기(Undo), 괄호 검증에 사용됩니다.

Queue(큐)는 선입선출(FIFO) 구조입니다. enqueue로 넣고 dequeue로 꺼냅니다. BFS, 작업 스케줄링, iOS 메시지 처리에 사용됩니다.

Swift에서는 protocol로 추상화하면 Array/LinkedList 등 다양한 내부 구현을 교체할 수 있습니다.

시간 복잡도
연산StackQueue (Array)Queue (LinkedList)
push / enqueueO(1)O(1) amortizedO(1)
pop / dequeueO(1)O(n)O(1)
peekO(1)O(1)O(1)

Chapter 04
Deque
양쪽 모두 O(1) — 링 버퍼로 구현하는 Double-Ended Queue
개념

Deque(덱)은 앞(front)과 뒤(back) 양쪽에서 O(1)으로 삽입/삭제가 가능한 자료구조입니다. Stack과 Queue를 모두 대체할 수 있습니다.

링 버퍼(Circular Buffer)로 구현하면 고정 크기 배열에서 head/tail 인덱스만 이동하여 메모리 재할당 없이 O(1) 연산을 달성합니다. Swift의 Collections 패키지 Deque가 이 방식입니다.

시간 복잡도
연산링 버퍼 Deque
prepend / appendO(1) amortized
removeFirst / removeLastO(1)
인덱스 접근O(1)

Chapter 05
Binary Tree
전위 · 중위 · 후위 · 레벨 순회 — 트리의 모든 것
개념

이진 트리는 각 노드가 최대 2개의 자식(left, right)을 가지는 계층적 자료구조입니다. 재귀적 구조이기 때문에 재귀 알고리즘과 찰떡궁합입니다.

순회(Traversal) 4가지: 전위(Pre-order) 루트→좌→우, 중위(In-order) 좌→루트→우, 후위(Post-order) 좌→우→루트, 레벨 순서(BFS) 층별 탐색.

순회별 활용
  1. 전위 — 트리 복사, 직렬화
  2. 중위 — BST에서 정렬된 순서 출력
  3. 후위 — 하위 폴더 크기 계산, 메모리 해제
  4. 레벨 BFS — 최단 경로, 층별 출력
생각해볼 것

체크리스트

중위 순회를 Stack 기반 반복문으로 구현했다
트리의 높이와 노드 수를 재귀로 구했다
Chapter 06
Binary Search Tree
삽입 · 탐색 · 삭제(3케이스) — 정렬된 트리의 힘
개념

BST는 모든 노드가 왼쪽 자식 < 현재 노드 < 오른쪽 자식 규칙을 만족하는 이진 트리입니다. 이 성질 덕분에 탐색이 절반씩 줄어들어 평균 O(log n)입니다.

삭제 3케이스: ① 자식 없음 — 그냥 삭제, ② 자식 1개 — 자식으로 대체, ③ 자식 2개 — 오른쪽 서브트리의 최솟값(Successor)으로 대체.

시간 복잡도
연산평균 (균형)최악 (편향)
탐색O(log n)O(n)
삽입O(log n)O(n)
삭제O(log n)O(n)

Chapter 07
Heap / Priority Queue
siftUp · siftDown — 가장 크거나 작은 것을 항상 O(1)에
개념

Heap은 완전 이진 트리를 배열에 저장하는 자료구조입니다. Max Heap은 부모가 자식보다 항상 크거나 같고, Min Heap은 그 반대입니다.

인덱스 i의 부모: (i-1)/2, 왼쪽 자식: 2*i+1, 오른쪽 자식: 2*i+2.

Priority Queue는 Heap으로 구현됩니다. iOS의 OperationQueue.qualityOfService, Dijkstra 알고리즘에서 핵심 역할.

시간 복잡도
연산복잡도
최대/최소값 조회 (peek)O(1)
삽입 (siftUp)O(log n)
삭제 (siftDown)O(log n)
Heapify (배열→힙)O(n)

Chapter 08
Graph
인접행렬 vs 인접리스트 · BFS · DFS — 관계를 모델링하다
개념

그래프는 정점(Vertex)과 간선(Edge)으로 구성됩니다. 방향/무방향, 가중치 유무로 다양한 종류가 있습니다.

인접행렬: O(1) 간선 확인, O(V²) 공간 — 밀집 그래프에 유리. 인접리스트: O(degree) 간선 확인, O(V+E) 공간 — 희소 그래프에 유리.

BFS는 Queue 사용, 최단 경로(가중치 없는). DFS는 Stack/재귀 사용, 사이클 감지·위상정렬.

시간 복잡도
표현 방식공간간선 확인인접 목록
인접행렬O(V²)O(1)O(V)
인접리스트O(V+E)O(degree)O(degree)
체크리스트
BFS로 최단 경로(홉 수)를 구했다
DFS로 사이클 감지를 구현했다

Chapter 09
Hash Table
충돌 해결 · Swift Dictionary 내부 — O(1) 탐색의 비밀
개념

Hash Table은 키를 해시 함수로 인덱스로 변환하여 O(1) 삽입/탐색/삭제를 달성합니다.

충돌 해결: ① Chaining — 같은 버킷에 LinkedList 연결 (최악 O(n)). ② Open Addressing — 빈 슬롯을 찾아 이동 (캐시 친화적). Swift Dictionary는 Open Addressing + Robin Hood Hashing을 사용합니다.

Load Factor가 임계값(보통 0.75)을 넘으면 버킷을 2배로 늘리고 Rehashing합니다.

Chapter 10
Trie
자동완성 DFS · O(m) 탐색 — 문자열 특화 트리
개념

Trie(트라이)는 문자열을 글자 단위로 트리에 저장하는 자료구조입니다. 검색 길이가 m이면 O(m)으로 탐색 가능 — 해시 충돌 없이 정확합니다.

각 노드는 자식 딕셔너리([Character: TrieNode])와 isEndOfWord 플래그를 가집니다. 자동완성은 prefix까지 내려간 뒤 DFS로 모든 단어를 수집합니다.

실제 사용처: iOS 키보드 자동완성, 검색 엔진 prefix 인덱싱, IP 라우팅 테이블.

시간 복잡도
연산복잡도비고
삽입O(m)m = 문자열 길이
탐색O(m)정확히 m 노드만 방문
자동완성O(m + k)k = 결과 수
공간O(알파벳 × 노드 수)영문 최대 26배

Chapter 11
AVL Tree
높이 균형 보장 — 회전(Rotation)으로 O(log n) 유지
개념

AVL Tree는 모든 노드에서 왼쪽/오른쪽 서브트리 높이 차이(Balance Factor)가 최대 1인 자기 균형 BST입니다. 1962년 Adelson-Velsky와 Landis가 고안했습니다.

Balance Factor = height(left) - height(right). BF가 ±2가 되면 회전(Rotation)으로 균형을 복원합니다.

4가지 회전 케이스: LL(우회전), RR(좌회전), LR(좌회전 후 우회전), RL(우회전 후 좌회전).

회전 케이스 정리
  1. LL — 왼쪽 자식의 왼쪽에 삽입 → 우회전(Right Rotation)
  2. RR — 오른쪽 자식의 오른쪽에 삽입 → 좌회전(Left Rotation)
  3. LR — 왼쪽 자식의 오른쪽에 삽입 → 좌회전 후 우회전
  4. RL — 오른쪽 자식의 왼쪽에 삽입 → 우회전 후 좌회전
시간 복잡도
연산평균최악
탐색 / 삽입 / 삭제O(log n)O(log n)
회전O(1)O(1)

균형이 보장되므로 일반 BST와 달리 최악도 O(log n)입니다.

생각해볼 것
AVL Tree는 삽입/삭제마다 회전이 발생할 수 있어 쓰기가 잦은 경우 Red-Black Tree보다 느릴 수 있습니다. 읽기:쓰기 비율이 어떨 때 AVL이 유리할까요?
체크리스트
Balance Factor를 계산하고 4가지 회전을 구현했다
삽입 후 균형 복원 과정을 단계별로 추적했다
일반 BST와 AVL Tree의 높이를 같은 입력으로 비교했다
삭제 후 재균형 로직을 구현했다
Chapter 12
Red-Black Tree
5가지 규칙으로 느슨한 균형 — Swift의 SortedSet 내부
개념

Red-Black Tree는 각 노드가 Red/Black 색상을 가지며 5가지 규칙으로 높이를 O(log n)으로 보장하는 균형 BST입니다.

5가지 규칙:

  1. 모든 노드는 Red 또는 Black
  2. 루트는 항상 Black
  3. 모든 리프(NIL)는 Black
  4. Red 노드의 두 자식은 항상 Black
  5. 임의의 노드에서 리프까지 경로의 Black 노드 수는 동일 (Black-height)

Swift의 OrderedSet, C++의 std::map, Java의 TreeMap이 Red-Black Tree로 구현됩니다.

AVL vs Red-Black 비교
항목AVL TreeRed-Black Tree
균형 엄격도엄격 (BF ≤ 1)느슨 (Black-height)
탐색 속도더 빠름약간 느림
삽입/삭제더 많은 회전더 적은 회전
최악 높이~1.44 log n~2 log n
적합한 상황읽기 빈번쓰기 빈번
생각해볼 것
Red-Black Tree의 삽입 후 규칙 복원(Fixup)은 최대 몇 번의 회전을 수행할까요? AVL Tree와 비교해보세요.
체크리스트
5가지 규칙을 외우고 각각의 의미를 설명할 수 있다
삽입 후 Recoloring과 Rotation을 적용하는 과정을 이해했다
Swift OrderedCollections 패키지 소스를 읽어봤다

Segment Tree
구간 쿼리 O(log n) — 합계 · 최솟값 · 최댓값 즉시 조회
개념

세그먼트 트리는 배열의 구간(range) 쿼리를 빠르게 처리하는 트리입니다. 각 노드가 특정 구간의 집계값(합/최솟값/최댓값 등)을 저장합니다.

배열 크기 n에 대해 2n 크기의 배열로 표현할 수 있습니다 (Heap과 유사한 인덱스 구조). 리프가 원소, 내부 노드가 구간 집계값입니다.

구간 업데이트가 잦을 때는 Lazy Propagation으로 O(log n) 범위 업데이트가 가능합니다.

시간 복잡도

단일 원소 업데이트O(1)O(log n)
구간 업데이트 (Lazy)O(n)O(log n)
빌드—O(n)

Fenwick Tree(Binary Indexed Tree)는 Segment Tree의 서브셋 기능만 제공하지만 코드가 훨씬 간결합니다. 두 자료구조의 트레이드오프를 비교해보세요.
체크리스트

Chapter 14
Union-Find (Disjoint Set)
연결 여부를 거의 O(1)에 — Kruskal MST의 핵심
개념

Union-Find는 원소들을 분리된 집합(Disjoint Set)으로 관리하며 두 연산을 지원합니다: find(x) — x가 속한 집합의 대표(root) 반환, union(x, y) — x와 y의 집합을 합침.

Path Compression: find 시 경유 노드를 모두 root에 직접 연결 → 다음 find가 O(1)에 가까워짐.

Union by Rank: 항상 낮은 트리를 높은 트리 아래에 붙임 → 트리 높이 O(log n) 보장.

두 최적화를 합치면 사실상 O(α(n)) (역 아커만 함수, 상수에 가까움).

실전 활용
  1. Kruskal MST — 간선 추가 시 사이클 감지에 Union-Find 사용
  2. 네트워크 연결 확인 — 두 서버가 같은 클러스터인지 O(α) 판별
  3. Percolation — 격자가 연결됐는지 시뮬레이션
시간 복잡도
연산단순 구현Path Compression + Rank
findO(n)O(α(n)) ≈ O(1)
unionO(n)O(α(n)) ≈ O(1)
체크리스트
Path Compression을 적용한 find를 구현했다
Union by Rank를 적용한 union을 구현했다
Kruskal 알고리즘에서 Union-Find를 활용했다
섬의 개수 세기, 친구 그룹 문제를 풀었다

Algorithm 01
Sorting
Quick · Merge · Heap — 각각의 장단점과 선택 기준
개념

Quick Sort: pivot을 기준으로 분할 정복. 평균 O(n log n), 최악 O(n²). 캐시 친화적이라 실제로 가장 빠릅니다. Swift의 sort()는 IntroSort(Quick+Heap+Insertion 혼합).

Merge Sort: 항상 O(n log n), 안정 정렬(Stable). 추가 O(n) 메모리 필요. 연결 리스트 정렬에 최적.

Heap Sort: 항상 O(n log n), 추가 메모리 O(1). 실제로는 캐시 미스로 Merge Sort보다 느린 경우 많음.

비교
알고리즘평균최악공간안정
Quick SortO(n log n)O(n²)O(log n)✗
Merge SortO(n log n)O(n log n)O(n)✓
Heap SortO(n log n)O(n log n)O(1)✗
Insertion SortO(n²)O(n²)O(1)✓
Counting SortO(n+k)O(n+k)O(k)✓
생각해볼 것
Swift의 Array.sort()가 IntroSort를 쓰는 이유는 무엇인가요? 작은 배열(≤16)에서 Insertion Sort로 전환하는 이유는?
체크리스트
Quick Sort의 pivot 선택 전략 3가지를 알고 있다 (first/random/median-of-3)
Merge Sort를 구현하고 안정 정렬인지 확인했다
정렬된 배열/역정렬 배열에서 Quick Sort 최악 케이스를 실험했다
면접에서 "어떤 정렬을 쓰시겠어요?"에 이유와 함께 답할 수 있다
Algorithm 02
Binary Search
이분 탐색 · Lower/Upper Bound — 정렬 배열에서 O(log n)
개념

이분 탐색은 정렬된 배열에서 중간값과 비교하며 탐색 범위를 절반씩 줄입니다. 핵심은 left, right, mid 인덱스 관리와 종료 조건입니다.

Lower Bound: target 이상의 첫 번째 인덱스. Upper Bound: target 초과의 첫 번째 인덱스. 두 값의 차이 = target의 개수.

이분 탐색은 단순 값 찾기 외에도 "조건을 만족하는 최솟값/최댓값" 탐색에 강력합니다 (Parametric Search).

off-by-one 방지 패턴
  1. while left < right — left == right일 때 종료 (Lower Bound용)
  2. while left <= right — 정확한 값 탐색용
  3. mid = left + (right - left) / 2 — 오버플로우 방지
  4. 조건에 따라 right = mid vs right = mid - 1 선택
시간 복잡도
연산복잡도
탐색 (정렬 배열)O(log n)
Lower/Upper BoundO(log n)
Parametric SearchO(log(범위) × 검증비용)
체크리스트
off-by-one 없는 Binary Search를 암기했다
Swift로 Lower Bound / Upper Bound를 직접 구현했다
"최소 몇 일?"류의 Parametric Search 문제를 풀었다
회전된 정렬 배열에서 탐색 문제를 풀었다
Algorithm 03
Dynamic Programming
중복 부분 문제 + 최적 부분 구조 — 지수를 다항으로
개념

DP의 적용 조건 두 가지: ① 최적 부분 구조 — 큰 문제의 최적해가 작은 문제의 최적해로 구성됨. ② 중복 부분 문제 — 같은 하위 문제가 여러 번 반복됨.

Top-down (Memoization): 재귀 + 캐시 딕셔너리. 직관적이나 스택 오버플로 주의.

Bottom-up (Tabulation): 작은 문제부터 테이블 채우기. 반복문, 공간 최적화 쉬움.

자주 나오는 DP 유형
  1. 1D DP — 피보나치, 계단 오르기, 동전 거스름돈
  2. 2D DP — 최장 공통 부분수열(LCS), 배낭 문제(Knapsack)
  3. 구간 DP — 행렬 연쇄 곱셈, 풍선 터뜨리기
  4. LIS — 최장 증가 부분수열 O(n log n)
  5. 경로 DP — 격자 최솟값 경로
생각해볼 것
DP와 Greedy는 모두 최적화 문제에 쓰입니다. Greedy가 통하지 않는 경우(0-1 배낭 문제 등)와 DP를 써야 하는 이유를 예시로 설명해보세요.
체크리스트
피보나치를 재귀→Memoization→Tabulation 3가지로 구현했다
LCS를 2D 테이블로 풀었다
LIS를 O(n²)과 O(n log n) 두 가지로 구현했다
0-1 배낭 문제를 DP로 풀었다
Algorithm 04
Greedy
지금 최선이 전체 최선 — 증명이 핵심
개념

Greedy 알고리즘은 매 단계에서 지역적으로 최선인 선택을 합니다. 전역 최적해가 보장되려면 Greedy Choice Property와 최적 부분 구조가 성립해야 합니다.

증명 방법: Exchange Argument — Greedy 선택을 다른 선택으로 바꿔도 결과가 나빠지지 않음을 보임.

대표 Greedy 문제
  1. Activity Selection — 종료 시간 순 정렬 후 겹치지 않게 선택
  2. Fractional Knapsack — 단위 가치(가치/무게) 순 정렬 후 채우기
  3. Huffman Coding — 빈도 낮은 것부터 Min Heap으로 합치기
  4. Kruskal MST — 가중치 낮은 간선부터 Union-Find로 추가
  5. 거스름돈 — 큰 동전부터 (단, 동전 종류에 따라 Greedy가 틀릴 수 있음)
생각해볼 것
동전이 [1, 5, 6, 9]원이고 11원을 거슬러야 할 때 Greedy(9+1+1=3개)는 최적이 아닙니다(5+6=2개). 어떤 조건에서 Greedy가 틀리나요?
체크리스트
Activity Selection을 구현하고 Greedy가 최적임을 증명했다
Huffman Coding의 원리를 이해하고 트리를 그려봤다
Greedy가 틀리는 반례를 직접 찾아봤다
Algorithm 05
Backtracking
가지치기로 완전 탐색 최적화 — N-Queens · 순열 · 스도쿠
개념

백트래킹은 가능한 모든 경우를 탐색하되, 해가 될 수 없는 경우를 조기에 포기(Pruning)하는 전략입니다. DFS 기반으로 동작합니다.

패턴: 선택 → 제약 확인 → 재귀 → 선택 취소(Undo). 상태를 선택 전으로 되돌리는 것이 핵심입니다.

최악 시간복잡도는 완전 탐색과 같지만, Pruning이 효과적이면 실제 탐색 공간이 크게 줄어듭니다.

대표 Backtracking 문제
  1. N-Queens — 체스판에 N개의 퀸 배치 (행/열/대각선 체크)
  2. 순열/조합 — 중복 여부와 순서 유무로 4가지 유형 정리
  3. 스도쿠 풀기 — 빈 칸에 1~9 시도, 충돌 시 백트래킹
  4. 단어 검색 (Word Search) — 격자에서 DFS로 단어 탐색
체크리스트
N-Queens를 완전히 구현하고 동작을 단계별로 추적했다
순열/조합/중복순열/중복조합 4가지를 모두 구현했다
Pruning으로 탐색 공간이 줄어드는 것을 카운터로 측정했다
Algorithm 06
최단 경로
Dijkstra · Bellman-Ford · Floyd-Warshall
개념

Dijkstra: 음수 가중치 없는 그래프, 단일 출발점. Priority Queue 사용 O((V+E) log V). 실제 내비게이션, 라우팅 프로토콜에 사용.

Bellman-Ford: 음수 가중치 허용, 음수 사이클 감지 가능. O(VE)로 느리지만 범용적. V-1번 모든 간선 Relax.

Floyd-Warshall: 모든 쌍(All-pairs) 최단 경로. O(V³), 음수 가중치 허용, 소규모 밀집 그래프에 적합.

비교
알고리즘시간음수 가중치음수 사이클용도
DijkstraO((V+E) log V)✗✗단일 출발
Bellman-FordO(VE)✓감지 ✓단일 출발
Floyd-WarshallO(V³)✓감지 ✓모든 쌍
생각해볼 것
Dijkstra에서 Priority Queue 대신 단순 배열을 쓰면 O(V²)이 됩니다. V가 크고 E가 적은 희소 그래프 vs V가 작고 E가 많은 밀집 그래프에서 각각 어느 쪽이 유리할까요?
체크리스트
Priority Queue 기반 Dijkstra를 구현했다
Bellman-Ford로 음수 사이클을 감지했다
Floyd-Warshall의 점화식 dp[i][j] = min(dp[i][j], dp[i][k]+dp[k][j])를 이해했다
문제를 보고 세 알고리즘 중 어떤 것을 쓸지 판단할 수 있다
